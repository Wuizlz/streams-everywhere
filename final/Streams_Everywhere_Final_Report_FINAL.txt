Streams Everywhere: Comparing Local Pipelines and HTTP in Java

Daniel Briseno
CS 33600: Network Programming
Instructor: Roger Kraft
Spring 2026

Abstract

This paper compares two forms of communication in Java that both rely on
the idea of streams but operate under very different conditions: local
inter-process communication (IPC) with ProcessBuilder and network
communication with HttpClient. The project asks how communication
between processes on the same computer compares to communication over
HTTP, and what those differences reveal about isolation, correctness,
reliability, and observability. To answer that question, the project
uses two small demos. The IPC demo launches child processes, captures
stdout and stderr, builds a simple pipeline, redirects output to a file,
and records exit codes. The HTTP demo sends GET requests, reads status
codes and headers, captures response bodies as raw bytes, and decodes
text using the charset declared in Content-Type. Runtime evidence from
the demos shows that both models are stream-oriented, but they
communicate failure and state in different ways. Local IPC gives the
parent process direct control over child streams and exit codes, while
HTTP adds protocol metadata such as status codes and headers. HTTPS can
also introduce a trust layer before any HTTP response is available. The
result is that similar stream mechanics can hide very different
debugging, reliability, and correctness concerns.

Introduction

Streams are one of the most common abstractions in programming because
many systems ultimately move data from one place to another in sequence.
Java uses streams when a parent process reads a child process’s stdout,
when two programs are connected with a pipe, and when a client reads an
HTTP response body from a remote server. At a high level, these
situations can look similar: bytes are produced, transmitted, and
consumed. However, that similarity is only partial. The surrounding
control model, error model, and debugging tools are very different
depending on whether communication stays on the local machine or crosses
a network.

This paper argues that Java uses streams in both local and networked
communication, but developers need different mental models for each
case. Local IPC with ProcessBuilder is closely tied to process creation,
standard streams, exit codes, and operating-system level redirection.
HTTP communication with HttpClient adds application protocol semantics
such as request methods, status codes, response headers, and text
decoding rules. If HTTPS is involved, the request can even fail before
any HTTP status code is returned because the TLS certificate chain must
be trusted first. These differences matter because they shape how
developers observe failures, reason about correctness, and choose
debugging strategies.

The rest of the paper proceeds in five parts. First, it explains the
background concepts behind Java streams, local process communication,
and HTTP request and response handling. Second, it analyzes the IPC demo
and its use of inherited terminal I/O, captured streams, pipelines, file
redirection, and exit codes. Third, it analyzes the HTTP demo and its
handling of status codes, headers, response bytes, charsets, redirects,
forbidden responses, and HTTPS trust failures. Fourth, it compares the
two models directly. Finally, it summarizes what the demos show about
practical software development in Java.

Background

At a general level, a stream is an ordered flow of data that can be read
or written incrementally. Java programmers encounter that idea in many
APIs, from InputStream and OutputStream to network sockets and HTTP
response bodies. The abstraction is useful because it separates how data
is produced from how it is consumed. A program can read text from a
file, from another process, or from an HTTP server using related
patterns even when the sources are very different. That said, the
surrounding environment still matters. A local child process and a
remote web server do not offer the same guarantees or the same failure
modes.

For local IPC, Java’s ProcessBuilder API creates new operating-system
processes and lets the parent process decide what happens to stdin,
stdout, and stderr (Oracle, n.d.-a). Those streams can be inherited by
the parent terminal, captured programmatically, redirected to files, or
connected in a pipeline. The parent can also wait for the child to
finish and inspect the resulting exit code. These features make
ProcessBuilder a useful way to study stream-based communication on one
machine because it exposes both the data flow and the control flow of
child processes.

For network communication, Java’s HttpClient API builds requests and
receives responses that contain a status code, headers, and a body
(Oracle, n.d.-b; Oracle, n.d.-e). The body can be handled as bytes or
converted to text. Unlike local process communication, HTTP is not just
a raw data channel. It is a protocol with standardized semantics for
redirects, authentication challenges, content types, and status codes
(Fielding et al., 2022). A Java HTTP client must therefore interpret
both the stream of bytes and the protocol metadata around that stream.

Character encoding is another important background concept. If a
response body contains text, the client must decode bytes into
characters using the correct charset. The Content-Type header may
declare that charset, and Java’s Charset.forName(...) can resolve it at
runtime (Oracle, n.d.-g). Decoding with the wrong charset can still
produce output, but the content may be corrupted or misleading. This is
one reason the HTTP demo captures raw bytes first and decodes them only
after reading the response headers.

HTTPS adds one more layer before HTTP response handling even begins.
Before the client can safely read status codes or headers, the TLS
handshake must succeed. That requires the client to validate the server
certificate chain against a trusted root CA in its trust store. This
trust step has no direct equivalent in local process pipelines. As a
result, a network request can fail before the application reaches the
stage where it can inspect HTTP-level information.

The course materials give a useful visual bridge between these ideas.
The Streams and ProcessBuilder notes show local processes communicating
through standard input, standard output, standard error, redirection,
and pipes (Kraft, n.d.-d, pp. 2, 8-19; Kraft, n.d.-c, pp. 5-6, 16-17).
The Network Clients notes extend that picture to two hosts connected by
sockets, where each process uses input and output streams around a
bidirectional network connection (Kraft, n.d.-b, pp. 3-5). Finally, the
HTTP notes show that HTTP messages are structured as an initial request
or status line, headers, a blank-line separator, and an optional body
(Kraft, n.d.-a, pp. 3-5). These diagrams are useful evidence because
they support the paper’s main comparison: streams remain the common
mechanism, but the surrounding communication structure changes from
local operating-system objects to network sockets and then to HTTP
protocol messages.

Local IPC in Java

The IPC demo in this project uses ProcessBuilderIpcDemo together with
two small helper programs: MessageProducer and UppercaseFilter. The
producer writes three lines to stdout, emits a warning to stderr, and
exits with a code provided on the command line. The filter reads lines
from stdin and transforms them into numbered uppercase output. Although
the demo is small, it is enough to show several important behaviors of
local IPC in Java.

The first example uses inheritIO(). In that mode, the child process
writes directly to the same terminal as the parent. This is convenient
because it requires almost no extra code and looks natural to a user
watching the program run. It also makes the timing relationship between
stdout and stderr visible. In the demo output, the warning sent to
stderr appears intermixed with the normal stdout lines. That behavior is
realistic, but it is also harder to analyze afterward because stdout and
stderr are no longer separated. When the goal is observation rather than
raw convenience, inherited terminal I/O can be too noisy.

The second example captures stdout and stderr programmatically. Instead
of letting the child write directly to the console, the parent reads
both streams through Java input streams. This makes the output easier to
examine because normal output and error output can be printed in
separate labeled sections. The captured example in this project also
uses a nonzero exit code, which shows that stream content alone is not
enough to describe program success or failure. The parent needs both the
text produced by the child and the child’s exit code to interpret the
result correctly. This is an important difference from HTTP, where there
is no parent-managed exit code from the server process.

The third example uses ProcessBuilder.startPipeline(...) to connect two
child processes. The stdout of the producer becomes the stdin of the
uppercase filter. This is the clearest demonstration that local process
communication is truly stream-based. The first process does not know the
details of the second process. It simply writes data to stdout, and the
operating system plus Java’s pipeline setup deliver that stream to the
next process. The final output is a transformed version of the original
stream: alpha, bravo, and charlie become numbered uppercase lines. The
example is small, but it illustrates a core Unix-style idea that Java
can reproduce: one program produces a stream, another consumes it, and
the pipe itself defines the connection.

This behavior matches the course pipeline diagrams, where a pipe sits
between the producer’s stdout and the consumer’s stdin rather than
letting the two processes attach to each other directly (Kraft, n.d.-d,
pp. 17-19).

The fourth example redirects stdout to a file while leaving stderr
inherited by the terminal. This split is useful because it shows that
stdout and stderr are not just conceptually different; they can also be
routed to different destinations. In the demo, the normal output is
saved to producer-output.txt, while the warning still appears live on
the console. That setup is practical in real systems because it lets a
developer persist primary output while still seeing urgent error
messages immediately.

Taken together, these IPC behaviors show that local process
communication is deeply tied to parent-child control. The parent decides
how streams are wired, whether outputs are captured or inherited,
whether a pipeline exists, and when to wait for completion.
Observability is strong because the parent can inspect both data streams
and exit codes directly. At the same time, this model places more
responsibility on the programmer. If the parent does not consume child
streams carefully, buffering and blocking can become hidden sources of
trouble (Oracle, n.d.-a). Even in a simple demo, the developer must
think about stream destination, process order, and termination.

HTTP Streams in Java

The HTTP demo takes a different approach because the other endpoint is
not a child process under local control. Instead, the program creates a
request with HttpRequest.newBuilder(), sends it through HttpClient, and
receives an HttpResponse<byte[]> object (Oracle, n.d.-b; Oracle, n.d.-d;
Oracle, n.d.-e). The client is configured with
followRedirects(HttpClient.Redirect.NEVER), which means redirect
responses are preserved for inspection rather than handled automatically
(Oracle, n.d.-c). This choice is useful for the project because it keeps
3xx responses visible.

The body handler used in the demo is
HttpResponse.BodyHandlers.ofByteArray() (Oracle, n.d.-f). That design
choice matters because it separates transport from interpretation. The
client first receives the body as raw bytes, then looks at the
Content-Type header, extracts the charset if present, and only then
decodes the bytes into text. This is a more careful approach than
immediately assuming UTF-8 for every response. It also matches the
larger theme of the project: streams carry data, but correct
interpretation still depends on context.

The HTTP course notes reinforce this distinction by showing that the
body is only one part of the complete message; the request or status
line and headers arrive before the body and shape how the body should be
interpreted (Kraft, n.d.-a, pp. 3-5, 8-9). This is why the demo prints
status and selected headers before showing a decoded body preview.

The saved runtime evidence produced several useful examples. The runs in
evidence/http-google-output.txt, evidence/http-github-output.txt, and
evidence/http-example-output.txt show that requests to
https://www.google.com, https://api.github.com, and https://example.com
all returned 200 responses. The Google response declared
text/html; charset=ISO-8859-1, the GitHub API response declared
application/json; charset=utf-8, and the Example Domain response did not
declare a charset, so the demo used its UTF-8 fallback. These successful
cases show the normal request/response lifecycle: request creation, TLS
handshake, response status, header inspection, and body decoding.

The other saved runs show protocol-level variation. A request to
http://github.com, saved in evidence/http-redirect-output.txt, returned
a 301 response with a Location header pointing to https://github.com/. A
request to https://httpbin.org/status/403, saved in
evidence/http-403-output.txt, returned a 403 response. A request to
https://httpbin.org/basic-auth/user/passwd, saved in
evidence/http-401-output.txt, returned a 401 response with
WWW-Authenticate: Basic realm="Fake Realm". In all three cases, the Java
program still received an HTTP response object and could inspect the
status code and selected headers. Earlier TLS debug evidence in
handshake.log shows a different kind of failure, SSLHandshakeException
with PKIX path building failed, where Java could not build a trusted
certificate path. That earlier result is useful because it shows that
HTTPS adds an infrastructure dependency before HTTP semantics are
available. When the TLS handshake fails, there is no status code to
inspect and no response body to decode.

This observation deepens the contrast with local IPC. A child process
can fail, but the parent still typically receives some combination of
stderr, stdout, or an exit code. By contrast, network communication may
fail at multiple layers. Some failures are application-level, such as
401 Unauthorized or 403 Forbidden, which are still valid HTTP responses
with useful headers. Other failures occur earlier, such as TLS trust
validation errors, where the client never receives an HTTP response at
all. That layered behavior makes HTTP more expressive, but it also makes
debugging more complex.

The HTTP demo is therefore more than a basic fetch example. It shows
that a response body is only one piece of network communication. Correct
client logic also depends on redirect policy, header inspection, charset
selection, and the ability to distinguish transport, trust, and protocol
failures from one another. This layered model is one of the most
important differences between local pipelines and networked
communication.

Comparison

The clearest similarity between local IPC and HTTP is that both move
ordered data through streams. In both demos, the receiving side must
read data in the correct order, decide when enough data has been
received, and interpret what the bytes mean. Both systems also involve
control decisions outside the raw stream itself. A developer chooses
whether to inherit or capture a child process’s streams, just as a
developer chooses redirect policy or body handling for an HTTP client.

The major differences appear once control, failure, and observability
are considered. In local IPC, the parent process has direct authority
over the child process and its standard streams. The communication path
is short, local, and usually deterministic. Error signaling happens
through stderr and exit codes. In HTTP, the client and server are
separated by the network, and the server remains outside the client’s
direct control. Error signaling often happens through status codes and
headers, but failures can also occur earlier at the DNS, TCP, or TLS
layers.

Observability also differs. In local IPC, the parent can capture exact
child stdout and stderr and can decide how those streams are routed. In
HTTP, the client sees only what the protocol exposes. Status codes and
headers provide valuable metadata, but they do not replace direct
process visibility. The runtime evidence makes that point clear in two
ways: redirect and forbidden responses still provide HTTP metadata,
while the earlier TLS debug log shows that certificate-path validation
can fail before HTTP metadata exists at all.

The two models also differ in how they handle correctness. For local
pipelines, correctness often means connecting the right streams,
draining them safely, and checking exit codes. For HTTP, correctness
includes those stream concerns but adds protocol semantics and data
interpretation. A client must know whether to follow redirects, how to
parse headers, how to decode the response body, and how to interpret
status codes. If HTTPS is used, correctness also includes trusting the
certificate chain.

In practical terms, the comparison suggests that “both use streams” is
true but incomplete. Stream handling is the common mechanism, not the
complete model. Local IPC is closer to explicit orchestration of child
behavior. HTTP is closer to protocol-driven communication with
additional metadata and trust layers.

Demo Findings

The demos support three main conclusions. First, local IPC makes stream
routing and error visibility very concrete. Inherited terminal I/O is
simple but noisy. Captured stdout and stderr are much easier to analyze.
Pipelines show how one program’s output can become another program’s
input with little extra code. File redirection demonstrates that
standard streams can be separated and stored for later inspection. These
are strong examples of Java exposing classic process-level communication
patterns directly.

Second, the HTTP demo shows that response handling is shaped by
metadata, not just by raw bytes. Status codes, headers, and declared
charsets all influence what the response means. Reading the body as
bytes first and decoding later based on Content-Type is a defensible
design because it reduces the risk of misinterpreting text content. This
is especially useful when comparing responses from different servers,
which may not use the same encoding.

Third, the HTTP and HTTPS tests reveal important real-world
complications. The saved HTTP runs show successful HTTPS responses, a
redirect, a 401 authentication challenge, and a 403 response, while the
earlier TLS debug log records SSLHandshakeException and
PKIX path building failed. Together, these results show that network
communication is not only about sending requests and reading responses.
It also depends on protocol metadata, trust in certificate authorities,
and the particular trust store used by the runtime. These dependencies
have no true equivalent in a local process pipeline.

These findings strengthen the paper’s central claim. Java streams are a
useful unifying idea, but the surrounding systems are different enough
that developers must reason differently about them. Local IPC emphasizes
process wiring and exit codes. HTTP emphasizes protocol metadata, remote
uncertainty, and, in the case of HTTPS, certificate validation before
any response can be processed.

Conclusion

This project began with a simple question: how does communication
between local processes compare to communication over HTTP in Java? The
answer is that both systems use streams, but they do not use them in the
same operational context. Local IPC with ProcessBuilder is direct,
parent-controlled, and closely tied to stdout, stderr, redirection,
pipelines, and exit codes. HTTP with HttpClient also depends on streamed
data, but it adds status codes, headers, body decoding, redirects,
authentication challenges, and sometimes TLS trust failures before the
HTTP response is even visible.

The demos show why these distinctions matter. A programmer can often
debug local IPC by capturing streams and checking exit codes. Network
debugging usually requires reasoning across more than one layer:
transport reachability, TLS trust, protocol semantics, and body
interpretation. This does not mean one model is better than the other.
Instead, it means that a shared stream-based abstraction hides important
differences in control, observability, and reliability.

The broader lesson is practical. Developers should not treat all
stream-based communication as interchangeable. Java provides consistent
tools for reading and writing streamed data, but effective use of those
tools depends on understanding the communication model around them. For
local pipelines, that means careful management of child process I/O. For
HTTP, that means careful interpretation of status codes, headers,
encodings, and trust relationships. The stream abstraction is the common
foundation, but the mental model must change with the environment.
